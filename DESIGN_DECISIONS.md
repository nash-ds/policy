# Policy Proposal Processing API – Design Decisions

## 1. Project Overview

The objective of this project was to develop a RESTful backend application that simulates a simplified insurance policy proposal processing system using Spring Boot.

The application follows a layered architecture and stores all application data in-memory using Java Collections, as specified in the assignment. No relational or NoSQL database has been used.

The implementation focuses on clean code, separation of concerns, maintainability, and extensibility while adhering to the given business rules.

---

# 2. Project Architecture

The project follows a standard three-layer architecture.

```
                Client
                   │
             REST Controller
                   │
             Service Layer
                   │
         In-Memory Java Collections
```

### Controller Layer

Responsibilities:

* Expose REST endpoints.
* Receive HTTP requests.
* Accept request bodies.
* Delegate processing to the service layer.
* Return appropriate responses.

The controller layer intentionally contains no business logic.

---

### Service Layer

The service layer contains the complete business logic.

Responsibilities include:

* Customer validation
* Policy validation
* Policy submission
* Policy number generation
* Audit creation
* Duplicate checks
* Business rule enforcement

Keeping validations inside the service ensures that every entry point follows the same business rules.

---

### Storage Layer

Instead of a database, Java Collections are used.

```
Customer
Map<UUID, Customer>

Policy Proposal
Map<UUID, PolicyProposal>

Audit
List<Audit>
```

Since persistence was not required, Java Collections provide a lightweight repository-like implementation.

# 4. Implementation Approach

The application was implemented feature-by-feature rather than writing every model first.

The implementation order was:

1. Customer Module
2. Reference Master
3. Policy Proposal
4. Audit Module
5. Exception Handling
6. Testing

This approach allowed each module to be completed and verified independently before introducing dependencies.

---

# 5. Why HashMap?

HashMap was chosen because every entity is uniquely identified by a UUID.

Example:

```
Customer
UUID → Customer

Policy
UUID → PolicyProposal
```

Advantages:

* Average O(1) lookup
* O(1) insertion
* O(1) updates
* Direct retrieval using ID

If a List had been used, every lookup would require iterating through all elements.

Example:

```
for(Customer c : customers)
```

which becomes O(n).

Since almost every API retrieves data by ID, HashMap was the better choice.

---

# 6. Why UUID?

UUIDs were used because:

* They are globally unique.
* IDs can be generated without maintaining counters.
* They avoid collisions.
* They simulate production systems where IDs are generated by backend services.

Clients never send UUIDs while creating entities.

The backend generates them automatically.

---

# 7. Why ProposalRequest DTO?

The client should not control system-generated fields.

Therefore a ProposalRequest DTO was introduced.

Fields excluded from the request:

* Policy ID
* Status
* Policy Number
* Apply Date
* Maturity Date

These values are generated by the backend.

This prevents accidental or malicious modification of internal data.

---

# 8. Customer Design Decisions

The Customer entity stores only customer-related information.

Validation includes:

* Name mandatory
* Age between 18 and 65
* Valid phone number
* Valid email
* Valid PAN format
* Unique phone
* Unique email
* Unique PAN

The service generates the customer UUID.

---

# 9. Policy Design Decisions

The PolicyProposal model stores information after validation.

The proposal initially starts with

```
Status = Draft
```

Only after successful submission:

* Policy Number is generated.
* Status changes to Submitted.
* Audit record is created.

This closely resembles the lifecycle of an insurance proposal.

---

# 10. Policy Number Generation

Initially, policy numbers were considered unnecessary because UUID already uniquely identified a proposal.

However, policy numbers serve a different business purpose.

UUID

* Internal identifier

Policy Number

* Business identifier issued after approval

A customer receives a policy number, not an internal UUID.

Therefore policy numbers are generated only after successful submission.

---

# 11. Maturity Date

Although not required by the assignment, a maturity date was added.

```
Maturity Date =
Apply Date + Policy Term
```

Reason:

It allows future extension where policies automatically move to a "Completed" state after maturity.

---

# 12. Audit Design

Every successful submission generates an audit record.

Audit contains:

* Audit ID
* Policy ID
* Customer ID
* Module
* Action
* Timestamp

The audit module provides traceability of important business events.

---

# 13. Validation Strategy

Two types of validation were used.

### API Boundary Validation

The controller ensures the request body is received correctly and delegates processing.

### Business Validation

The service validates:

Customer

* Exists

Policy

* Valid term
* Valid payment frequency
* Premium
* Sum assured
* PAN requirement
* Nominee validation

Business validation is centralized inside the service layer.

---

# 14. Exception Handling

A custom exception

```
BusinessValidationException
```

was created.

A global exception handler converts exceptions into meaningful HTTP responses.

Benefits:

* Cleaner controllers
* Consistent error messages
* Proper HTTP status codes
* Easier debugging

---

# 15. Design Trade-offs

## In-Memory Storage

Advantages

* Simple implementation
* Fast execution
* No database configuration
* Easy testing

Disadvantages

* Data lost after restart
* No persistence
* No transactions
* Limited scalability

---

## HashMap vs ConcurrentHashMap

HashMap was used for simplicity.

ConcurrentHashMap would be a better choice for production since it supports concurrent modifications safely.

ConcurrentHashMap was later implemented for thread safety

---

## Interfaces

Interfaces were introduced even though only one implementation exists.

Advantages

* Loose coupling
* Easier testing
* Easy replacement of implementation
* Better scalability

---

# 16. Assumptions

The following assumptions were made.

* Customer name is not unique.
* Phone number is unique.
* Email is unique.
* PAN is unique if provided.
* Customer must exist before creating a proposal.
* One proposal belongs to one customer.
* One policy number belongs to one proposal.
* Policy numbers are generated only after submission.
* Data remains only while the application is running.

---

# 17. Challenges Faced and Corrections Made

## 1. Customer ID was not stored

### Initial implementation

```
customers.put(UUID.randomUUID(), customer);
```

The generated UUID was only used as the HashMap key.

The Customer object itself did not store the generated ID.

Result:

Returned customer had a null customerId.

### Correction

Generate the UUID first.

```
UUID id = UUID.randomUUID();
customer.setCustomerId(id);
customers.put(id, customer);
```

Now both the HashMap key and the object contain the same ID.

---

## 2. Updating customer removed customer ID

Initially

```
customers.put(id, customer);
```

replaced the object.

Since the request body did not contain the UUID, the stored customer lost its ID.

### Correction

```
customer.setCustomerId(id);
```

before storing the object.

---

## 3. Duplicate validation failed during update

Initially, uniqueness checks iterated over every customer.

The customer being updated matched itself, causing false duplicate errors.

### Correction

While checking uniqueness, skip the customer currently being updated by comparing its ID.

This allows updating existing records without triggering duplicate validation against themselves.

---

## 4. Policy Submission Validation

Initially, validation accepted only ProposalRequest.

During submission, a PolicyProposal object already existed.

### Correction

An overloaded validation method was created.

```
validatePolicy(ProposalRequest)

validatePolicy(PolicyProposal)
```

Both reuse the same helper validation methods.

---

## 5. Large Validation Method

Initially, all validations were inside one large method.

Problems:

* Difficult to read
* Difficult to debug
* Difficult to extend

### Correction

Validation was split into smaller methods.

Examples:

* validateCustomer()
* validatePolicyName()
* validatePremium()
* validateNominee()
* validatePolicyTerm()
* validatePaymentFrequency()

This improved readability and maintainability.

---

## 6. Payment Frequency Validation Failed

Initially, the comparison failed because of inconsistent letter casing between the input and the reference data.

### Correction

Both values are normalized before comparison to ensure case-insensitive validation.

---

## 7. Constructor Injection

Initially, service dependencies were created manually using `new`.

Problems:

* Not managed by Spring
* Harder to test
* Breaks dependency injection

### Correction

Constructor injection with Spring-managed beans was adopted (using constructor injection/Lombok), allowing Spring to provide dependencies automatically.

---

## 8. Audit Generation

Initially, proposal submission only updated the status.

### Correction

Submission now also:

* Generates a policy number
* Creates an audit record
* Stores timestamp
* Stores module and action

This better reflects a real-world policy issuance workflow.

---

# 19. Future Improvements

If this application were extended further, the following improvements would be considered:

* Replace in-memory storage with a relational database using Spring Data JPA.
* Introduce authentication and role-based authorization.
* Add pagination and filtering for list APIs.
* Implement structured logging and monitoring.
* Generate richer OpenAPI documentation.
* Introduce repository abstractions to simplify migration to a database-backed implementation.
* Add support for policy cancellation, renewal, and automatic maturity processing.

# 20. Additional Features Implemented

In addition to the mandatory assignment requirements, the following enhancements were implemented to improve the application's design, maintainability, and realism:

* Automatically calculate the policy maturity date based on the application date and selected policy term.
* Generate a unique policy number only after successful proposal submission to simulate real-world policy issuance.
* Use UUIDs for customer, policy, and audit identifiers to ensure globally unique IDs.
* Introduce a `ProposalRequest` DTO to prevent clients from modifying backend-managed fields such as policy ID, status, policy number, and dates.
* Enforce additional customer validations including unique phone number, email address, and PAN number.
* Handle customer updates without triggering false duplicate validation errors by excluding the current customer during uniqueness checks.
* Generate detailed audit records containing audit ID, policy ID, customer ID, action, module, and timestamp for better traceability.
* Integrate Swagger/OpenAPI for interactive API documentation and easier testing during development.(http://localhost:8080/swagger-ui/index.html#/)
